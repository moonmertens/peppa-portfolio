# Plan: Chunk 1 — Project Scaffolding & Infrastructure

## Summary

Bootstrap the artist portfolio as a Next.js 16 App Router project with TypeScript, Tailwind CSS, and Sanity Studio v3 embedded at `/studio` via `next-sanity`. The result is a running dev server with a bare layout shell and a functioning Sanity Studio route, wired to a real Sanity project via environment variables.

---

## Specifications

### Functional Requirements

- `npm run dev` starts a local server at `http://localhost:3000`.
- `http://localhost:3000/studio` serves Sanity Studio (embedded, not a separate process).
- All pages render without runtime errors.
- TypeScript compiles without errors (`tsc --noEmit`).
- Tailwind CSS is configured and its utility classes are available to all components.
- Environment variables for Sanity (project ID, dataset, API version, read token) are read from `.env.local` and never committed to version control.

### Non-Functional Requirements

- Node.js >= 20 (LTS).
- All configuration files use TypeScript (`.ts`) not `.js` where the toolchain supports it.
- No placeholder pages with real content — only structural shell components.
- `.env.local` is listed in `.gitignore`.
- Sanity Studio is accessible only when the correct `SANITY_API_READ_TOKEN` is present (it will be restricted to logged-in Sanity users by Sanity's own auth by default).

### Acceptance Criteria

- `npm run dev` exits without errors; browser shows a blank/skeleton page at `/`.
- `http://localhost:3000/studio` loads the Sanity Studio login/home screen.
- `npx tsc --noEmit` exits with code 0.
- `npm run lint` exits with code 0.
- `.env.local` does not appear in `git status`.
- The folder structure matches the layout specified in this plan.

## Architecture

### Component Overview

| Module | Responsibility |
|---|---|
| `app/layout.tsx` | Root HTML shell: `<html>`, `<body>`, font variables, global CSS import |
| `app/page.tsx` | Homepage stub — renders nothing but confirms routing works |
| `app/(studio)/studio/[[...tool]]/page.tsx` | Catch-all route that renders the embedded Sanity Studio |
| `sanity/sanity.config.ts` | Sanity Studio configuration (project ID, dataset, plugins) |
| `sanity/sanity.client.ts` | `@sanity/client` instance used by Next.js server components |
| `sanity/env.ts` | Typed accessors for Sanity environment variables |
| `lib/` | Reserved for shared helpers (empty at this stage) |
| `components/` | Reserved for shared UI components (empty at this stage) |
| `tailwind.config.ts` | Tailwind theme extension (warm palette placeholders) |
| `app/globals.css` | Tailwind directives + CSS custom properties |

### Data Flow

At this chunk there is no real data flow. The wiring established here is:

```
.env.local
  └── sanity/env.ts          (reads + validates env vars)
        └── sanity/sanity.client.ts   (constructs @sanity/client)
              └── (server components in later chunks call this client)

sanity/sanity.config.ts
  └── app/(studio)/studio/[[...tool]]/page.tsx  (renders Studio via next-sanity)
```

## Implementation Steps

### Step 1 — Create the Next.js application

Run the following command from the `peppaportfolio/` directory. `create-next-app` will scaffold the project in the **current directory** (`.`), so it must be run from inside the repo root.

```bash
npx create-next-app@latest . \
  --typescript \
  --tailwind \
  --eslint \
  --app \
  --no-src-dir \
  --no-import-alias \
  --turbopack
```

Flag rationale:
- `--app` — enables App Router (required)
- `--typescript` — TypeScript throughout
- `--tailwind` — scaffolds `tailwind.config.ts` and `postcss.config.mjs`
- `--eslint` — scaffolds `.eslintrc.json`
- `--no-src-dir` — keep files at the root level (not inside a `src/` folder)
- `--no-import-alias` — skip `@/` alias; use explicit relative paths for clarity
- `--turbopack` — enables Turbopack for faster local dev

When prompted about the existing `CLAUDE.md`, select "keep existing file" / do not overwrite.

This generates:
- `package.json` with `next@^16`, `react@^19`, `react-dom@^19`, `typescript`, `@types/node`, `@types/react`, `@types/react-dom`, `tailwindcss@^4`, `postcss`, `eslint`, `eslint-config-next`
- `tsconfig.json`, `next.config.ts`, `tailwind.config.ts`, `postcss.config.mjs`
- `app/layout.tsx`, `app/page.tsx`, `app/globals.css`
- `.gitignore` (already includes `.env.local` and `.next/`)

### Step 4 — Create environment variable files

**`.env.local`** (never committed):

```
NEXT_PUBLIC_SANITY_PROJECT_ID=<your-project-id>
NEXT_PUBLIC_SANITY_DATASET=production
NEXT_PUBLIC_SANITY_API_VERSION=2024-01-01
SANITY_API_READ_TOKEN=<your-read-token>
```

- `NEXT_PUBLIC_` prefix exposes the variable to the browser (needed by the embedded Studio).
- `SANITY_API_READ_TOKEN` is a server-only token (no `NEXT_PUBLIC_` prefix) used for authenticated reads from server components in later chunks.

**`.env.example`** (committed):

export const client = createClient({
  projectId,
  dataset,
  apiVersion,
  useCdn: false, // set to true in production for public/non-authenticated data
})

export async function sanityFetch<T>(
  query: string,
  params: Record<string, unknown> = {}
): Promise<T> {

    // In server components, Next.js caching applies here

}

## Risks / Open Questions

1. **`create-next-app` prompt for existing files** — The scaffolder will detect `CLAUDE.md` and `PLAN.md` already in the directory. The coder agent must select "keep existing files" for both. The `--yes` flag skips this confirmation entirely, but it would overwrite existing files — do NOT use `--yes`.

2. **Tailwind v4 config syntax** — Tailwind CSS v4 (latest at time of writing: `4.3.0`) uses `@import "tailwindcss"` in CSS instead of the three `@tailwind` directives. `create-next-app` is aware of this and generates the correct CSS file; double-check if the scaffold generates the v3 format and update accordingly.

3. **Studio route group and root layout** — The `(studio)` route group does NOT create a segment in the URL. Its purpose is solely to scope a separate `layout.tsx` if needed. At this chunk, no `(studio)/layout.tsx` is created — the Studio renders inside the root `<html>/<body>` shell. If Sanity Studio has z-index or full-height issues in production, a dedicated layout (with `className="h-full"` on `<html>` and `<body>`) may be needed. This is a known pattern documented in next-sanity README.

4. **`dynamic = "force-dynamic"` on Studio page** — This directive prevents Next.js from statically pre-rendering the Studio route. It is required because the Studio references `window` during render and cannot be statically exported.

5. **Sanity project ID not yet known** — The actual project ID is obtained during Step 3. The coder agent should note that `.env.local` must be filled in before the dev server can start without env-var assertion errors.

6. **Node.js version** — Confirm the local environment runs Node.js >= 20. Run `node --version` before scaffolding.

7. **`@sanity/vision`** — This is bundled inside the `sanity` package; no separate install is needed. If a TypeScript error appears about missing types, import from `sanity` not `@sanity/vision`.
